# Findings log

Local working file. Not yet merged into `docs/submission.md` — do not treat this as the canonical record until that merge happens. Compiled from this project's VERIFY sessions (baseline, agent wrapper, judge node, FastAPI/Render deployment, Next.js/Vercel frontend) plus what `docs/submission.md` already records, so nothing observed along the way gets lost before the merge.

Layers: ingestion / retrieval / agent-tooling / generation / judge / API / frontend.

---

## 1. F1 — uncited fabrication hidden among cited sentences

**What happened:** Asked whether an umpire can reverse his decision (a genuine Laws-of-Cricket question, not in MYCA's documents). In 3/3 baseline+agent-wrapper trials (pre-judge, pre-F3-paragraph), the model called `search_rules_tool`, retrieved a tangentially related chunk (umpires as sole judges of light/weather), and added an uncited "it is generally accepted that..." claim, surrounded by properly-cited sentences. 2 of 3 trials at least flagged the rules didn't cover it first, but still added the fabrication anyway.
**Root cause:** No mechanism existed to check generated claims against retrieved evidence; the tangential chunk gave the model just enough material to rationalize speculating.
**Layer:** generation. **Status:** documented in `docs/submission.md`; addressed by the judge node (finding 7) and the F3 prompt paragraph (finding 3), but see finding 7 for a residual gap discovered during judge VERIFY.

## 2. F2 — multi-step arithmetic nondeterminism

**What happened:** A time-lost/overs-reduction calculation (the "33-minutes family") produced 3 correct answers and 1 wrong one across 4 attempts in baseline testing, with identical retrieved context and no fixed temperature. A related mid-innings weather-delay calculation was refused 3/3 times even though the needed formula was in the already-retrieved context.
**Root cause:** Single LLM call applying a multi-step numeric procedure is not deterministic; retrieval itself was reliable in both cases — the failure is purely in applying the retrieved rule.
**Layer:** generation. **Status:** documented in `docs/submission.md`; the judge catches wrong arithmetic (finding 8) rather than fixing generation, per this project's own scope decision (fixing generation is Task 6 evidence work).

## 3. F3 — general-cricket-knowledge scope gap

**What happened:** A base-Laws-of-Cricket question (e.g. umpire decision reversal, LBW definition) fits neither `search_rules_tool` (association-specific) nor `web_search_tool` (current public info) by description. The model filled the gap with speculation instead of an honest fallback.
**Root cause:** No routing guidance existed for this third category; MYCA's documents assume the base Laws rather than define them, so the corpus was never going to answer these regardless of retrieval quality.
**Layer:** generation (prompt). **Status:** fixed-at-generation — one new paragraph added to `AGENT_SYSTEM_PROMPT` instructing an honest "outside this association's documents" refusal for this category. Confirmed working live end-to-end (finding 18).

## 4. Tool-choice unreliability

**What happened:** Running the 33-minutes question 7 times before any fix: 1 of 7 runs never called `search_rules_tool` at all (model reasoned "this is a calculation question, not a rules question"). After hardening the tool's docstring (a calculation question still needs the tool first, to find the rule and its numbers), re-tested 5 times post-fix: 5/5 called the tool.
**Root cause:** The tool's description left room for the model to treat "needs arithmetic" as a reason to skip retrieval.
**Layer:** agent-tooling. **Status:** fixed-at-agent-tooling (docstring hardening), verified 5/5.

## 5. grade_scope cross-grade contamination

**What happened:** In that same 7-trial batch, 3 of 7 runs called the tool but left `grade_scope` unset (it was originally an LLM-controlled parameter), and cross-grade similarity search returned the Junior document's rule for a senior-grade question. Combined with finding 4's 1/7, only 3 of 7 trials retrieved from the correct document overall.
**Root cause:** Letting the model choose `grade_scope` reintroduced, one layer up, the exact cross-grade contamination risk the `match_rule_chunks` SQL filter was built to prevent.
**Layer:** agent-tooling. **Status:** fixed-at-agent-tooling — `grade_scope` removed from the tool's schema entirely, closure-captured at graph-build time from the trusted caller (same principle as `association_id`), invisible to and unchangeable by the model. Combined re-test with finding 4's fix: 5/5.

## 6. Golden eval set had fabricated/incorrect expected answers

**What happened:** Of the 14-row evaluation table, 5 rows had their expected answer corrected or confirmed fabricated against the actual source documents. Three specific cases: row 1 (expected "34 overs" was a flat-formula assumption; the real rule points to a lookup table, and 6 minutes lost is 33 overs, not 34), row 6 (expected answer described a powerplay rule; no MYCA document at any grade mentions powerplay at all — invented), row 14 (expected answer named a specific sightscreen/dead-ball rule; the real applicable rule is 4.9.12, a general boundary rule, not a specific one).
**Root cause:** Golden-set expected answers were drafted without full cross-checking against the ingested source documents.
**Layer:** ingestion (eval-set authoring, not the corpus itself). **Status:** fixed — corrected in the eval table.

## 7. Judge bypass on zero-tool-call turns

**What happened:** The judge node's original routing skipped judge evaluation whenever a turn used no tools (to exempt greetings). Once the F3 paragraph (finding 3) made the model often skip `search_rules_tool` for the umpire question, this same routing rule let a zero-tool-call turn's answer through unflagged. Re-running F1's umpire question 3x under this routing: 1 of 3 slipped an uncited fabrication through with no flag at all.
**Root cause:** "No tool used" was being used as a proxy for "this is a greeting," but a substantive answer can also skip tools.
**Layer:** judge (routing). **Status:** fixed-at-judge — routing narrowed to skip judge only for a true greeting (regex match on the human message: hi/hello/hey/thanks/bye and similar); every other non-tool-call answer now reaches the judge. Re-tested 3x after the fix: 0/3 unflagged fabrication (all three reached an acceptable end state — flagged, clean honest fallback, or recovered-via-retry).

## 8. Judge arithmetic catch rate

**What happened:** Across 4 observed arithmetic-bearing turns (3 explicit F2-family tests plus 1 that surfaced during clean-path testing), the judge caught wrong arithmetic 4/4 — it never let a bad calculation through unflagged, on either the first draft or the post-retry revision.
**Root cause / caveat:** The judge's own stated "correct" value disagreed across separate runs of the same question (e.g. 9 overs vs. 8 overs vs. different rounding logic) — it is reliably suspicious of unverifiable arithmetic, not reliably able to compute the right answer itself.
**Layer:** judge. **Status:** documented-for-Task-6 (catch-rate is a real, positive characterization; the judge's own arithmetic reliability is a separate open question, not fixed here by design — fixing generation-side arithmetic is out of this task's scope).

## 9. Judge fail-safe on judge-call failure

**What happened:** Deliberately pointed `JUDGE_MODEL` at an invalid model name (pre-deployment simulation). Confirmed: `judge_node`'s own exception handling caught the error, set `judge_error: True`, routed straight to the flagged terminal state without consuming the one retry, and surfaced the real proxy error in the reasoning field. `needs_human_review` came back `True`, never silently passed.
**Follow-up:** A later live-production anomaly (see finding 11) initially looked like this same error path firing (empty `flagged_claims`/`flagged_citations` is the error path's signature), but Render logs showed no `judge_node ERROR` lines in that window — it was a genuine FAIL verdict, not the error path. The error path itself has only been observed under deliberate simulation, not as a spontaneous production failure, to date.
**Layer:** judge. **Status:** fixed-at-judge, verified via simulation; the broader "never silently pass" property has also held in every genuine production case observed so far, via real FAIL verdicts rather than the error path.

## 10. Judge reasoning text contradicting its own structured verdict

**What happened:** One live, flagged response's `judge_reasoning` read as a full endorsement ("all claims are supported by the retrieved evidence... no unsupported claims or fabricated citations were found, and no arithmetic was present to validate") while the structured `verdict` field was still `FAIL` — which is what triggered the flag in the first place.
**Root cause:** The judge LLM's free-text reasoning and its own structured verdict field are two separate outputs from the same call and can disagree; routing logic correctly trusts only the structured `verdict` field, not the prose.
**Layer:** judge. **Status:** open-decision — behaviorally safe (the system flags rather than silently passing), but confusing to show a user reasoning text that reads as a clean pass next to a review banner. Reasoning should be read as a lead for a human reviewer, not as a verdict in itself.

## 11. Generator echoes large injected tool payloads verbatim

**What happened:** Observed on at least two separate occasions, in two different contexts: (a) after a retry injected broadened rule-search results into a `SystemMessage`, one draft's "answer" was literally the raw chunk-dump text verbatim, not synthesized prose; (b) in a separate live turn, after `web_search_tool` returned results, the draft answer was literally the raw web-search tool output dumped verbatim, including embedded dict-formatted metadata. Both times, the judge caught it and flagged the response — neither reached the user as if it were a real answer.
**Root cause:** When a large block of retrieved/injected material is placed in context with a revision instruction, the generator sometimes reproduces it verbatim instead of synthesizing a response citing it. Not confirmed whether this is retry-specific (the `SystemMessage` framing) or a general large-context-echo tendency — observed in both a first-draft (web tool) and a post-retry (rules broadened-context) case.
**Layer:** generation. **Status:** documented-for-Task-6 — a real generation-quality gap, but the judge's catch rate on it has been 2/2 so far; not fixed at generation (out of this project's judge-task scope).

## 12. ChatResponse verdict-field gap → item-zero fix

**What happened:** When a judge FAIL carried no `unsupported_claims`/`fabricated_citations` (e.g. an arithmetic-only failure, or the judge-error path before this fix), the API's `ChatResponse` gave the caller `needs_human_review: true` with both flagged-detail lists empty and no way to see why.
**Root cause:** The API response never exposed the judge's own `reasoning` string or its `arithmetic_ok` boolean, both of which exist in graph state on every judged turn.
**Layer:** API. **Status:** fixed-at-API — `judge_reasoning` (string, nullable) and `arithmetic_ok` (bool, nullable) added to `ChatResponse`, always populated from the last judge call regardless of PASS/FAIL, null only when the judge never ran. Verified live post-deployment.

## 13. Citation-extraction regex bug

**What happened:** When the model cited a bare `(Section 5.3.1)` without a document name (rather than the full `(Section X, Document)` format), the extraction regex — which assumed a comma-separated document always followed — kept matching past that citation's own closing paren, through unrelated prose, into a second citation's closing paren, producing one garbled citation spanning two unrelated ones. Observed once, live, immediately after the Render backend redeploy for item zero (finding 12).
**Root cause:** The regex's "document" capture group was mandatory and unbounded (`[^)]+` with no anchor), so a missing document trailing comma let it consume arbitrary intervening text.
**Layer:** API. **Status:** fixed-at-API — `section_number` bounded to word-characters/dots, `document` group made optional and `RuleCitation.document` made nullable end to end (Pydantic model, TS type, `CitationChip` component). Re-verified locally against the exact failing text and live in production afterward (including one message that correctly mixed a full citation with two bare ones).

## 14. Open-proxy exposure (caught in plan phase) → master_key fix

**What happened:** Planning the Render deployment, the LiteLLM proxy was going to get a public URL with no auth (`gateway/README.md` had explicitly noted "no master_key... fine for local single-developer use," not addressing deployment). An unauthenticated proxy on the public internet would be an open relay to the OpenAI billing account.
**Root cause:** The proxy's local-dev auth posture (none) was never revisited for the deployment step.
**Layer:** API (gateway). **Status:** fixed before deployment — `general_settings.master_key` added to `gateway/config.yaml` (env-sourced, inert when unset so local dev is unaffected), and the 3 `ChatOpenAI` call sites in `backend/agent.py` plus `backend/retrieval.py`'s embedding call updated to send `LITELLM_PROXY_API_KEY` when set.

## 15. Python-version mismatch on Render

**What happened:** First Render deploy failed: the platform defaulted to Python 3.14.3, and `litellm[proxy]==1.91.1` requires Python <3.14.
**Root cause:** No Python version was pinned anywhere in the deployment config; both local venvs were already on 3.11.9, but nothing told Render to match.
**Layer:** API (deployment config). **Status:** fixed — `PYTHON_VERSION: 3.11.9` added as a plain env var to both services in `render.yaml`, matching the exact version both local venvs run and every pinned dependency in both requirements files was already verified against.

## 16. Vercel Deployment Protection blocking public access

**What happened:** The first live Vercel URL shared for VERIFY redirected to `vercel.com/sso-api` — Deployment Protection was on, blocking any unauthenticated visitor (including curl-based testing, and any real user or grader).
**Root cause:** Default/inherited project setting; the URL given was also a deployment-specific hash URL rather than the stable production alias.
**Layer:** frontend (deployment config). **Status:** fixed — Deployment Protection disabled, stable production URL (`lexisport-cricket-rule-agent.vercel.app`) confirmed publicly reachable (200, no redirect) before VERIFY continued.

## 17. Cold-start timeout underestimate

**What happened:** Planned timeout budget assumed ~60-80s worst case (Render free-tier cold start + a retried judge-looped turn). Real measurement against the live deployment: a first request after 15+ minutes idle timed out at 90s on its own; a second request immediately after still took ~60s to fully resolve; only the third request was fully warm (~8s).
**Root cause:** The original estimate didn't account for a cold container boot possibly spanning more than one request before the service is genuinely warm.
**Layer:** frontend (proxy route). **Status:** fixed — proxy `maxDuration` raised 100s→170s, its internal fetch timeout 90s→150s, and the client-side fetch timeout 115s→190s (all still well under Vercel Hobby's confirmed 300s ceiling).

## 18. F3 layering working end-to-end live (positive result)

**What happened:** Live, a base-Laws question (LBW definition) correctly triggered the F3 honest-fallback wording ("that's general cricket knowledge outside this association's own rule documents..."). The model then added one extra unsupported elaboration beyond the pure refusal ("LBW dismissals are defined by the Laws of Cricket") — and the judge caught that too, flagging the response even though the core refusal behavior was exactly right.
**Root cause:** N/A — positive result. Two independent safety layers (the F3 prompt paragraph, and the judge) each did their job; the second layer caught what the first layer's otherwise-correct behavior didn't fully constrain.
**Layer:** generation + judge (combined). **Status:** confirmed working as designed, live.

## 19. Golden-set expected behavior for general-knowledge questions contradicts the deployed F3 fix

**What happened:** Building `evals/golden_set.json` (Task 5.1), the 4 `general_knowledge` items (base-Laws questions like "can an umpire reverse his decision," "what counts as LBW") were first drafted with `expected_behavior: honest_fallback` — matching the F3 behavior already confirmed working live (findings 3, 18). During VERIFY, re-reading `CLAUDE.md`'s own locked stack decision surfaced a direct conflict: it explicitly scopes Tavily to cover "public/current-info questions **and base Laws of Cricket questions the association's documents assume rather than define**" — meaning these same 4 questions were designed to get a real, sourced answer via web search (`web_answer`), not a bare refusal. All 4 items were changed to `expected_behavior: web_answer` to match the documented architecture.
**Root cause:** Two artifacts of this project independently describe the intended behavior for this exact question category, and they disagree: the deployed `AGENT_SYSTEM_PROMPT`'s F3 paragraph (confirmed working, findings 3/18) implements a pure honest-fallback refusal, while `CLAUDE.md`'s stack decision describes Tavily-backed answering for the same category. Neither was wrong when written — the F3 fix predates (or wasn't cross-checked against) the fuller Tavily-scope decision, or the Tavily-scope decision was never implemented back into the agent prompt/routing.
**Layer:** generation (prompt) vs. project docs (architecture decision) — a cross-artifact inconsistency, not a bug in either individual piece. **Status:** open, and important for the next task (4.2 harness) to know before running: scoring the current live agent against this golden set's 4 `general_knowledge` items will very likely show clean FAILs (or at least behavior mismatches) on all 4, not because the agent is broken, but because it still implements the pre-Tavily-scope-decision F3 design. Needs an explicit decision — update the agent to route these to `web_search_tool` per `CLAUDE.md`, or revert the golden set back to `honest_fallback` and treat `CLAUDE.md`'s Tavily line as aspirational/not-yet-implemented — before treating harness results on this subset as meaningful.

## 20. Golden-set authoring errors caught only by re-deriving from source, not by citation-checking alone

**What happened:** Two hand-written golden-set items passed an initial "does this section exist" citation check but were still substantively wrong, caught only when the user pushed back on the real-world plausibility of the answer:
- `eval-018` (Senior Women's rain-interruption overs reduction) originally answered "50 overs off" by reading a number straight out of the Appendix B table for a 200-minute loss. The citation was real and the table row was real, but Appendix B only governs late starts (Section 3.3.2) — mid-match rain uses a completely different formula (Section 5.3.2), and a 50-over reduction is impossible in a format capped at 20 overs total. The error survived a citation check because every individual fact (the table, the row, the number) was real; only checking the *scenario's plausibility against the grade's own overs cap* surfaced the problem.
- `eval-023` (Senior Women's fielding-side-short-of-overs penalty) asked about "the fielding side" finishing short without specifying which innings' fielding side — Section 5.3.3 (team bowling first) and Section 5.3.5 (team bowling second) impose two structurally different penalties (overs deducted from your own batting vs. runs awarded to the opponent) for what reads like the same scenario. The original item's number happened to be correct for one specific reading (team bowling second) but the question itself couldn't be answered unambiguously as phrased.
**Root cause:** My own citation-verification process (confirm the cited section exists, confirm the cited number appears in it) is necessary but not sufficient — it doesn't catch cases where the cited rule is the wrong rule for the scenario, or where the question itself conflates two branches of a grade's own rules that use different mechanisms.
**Layer:** ingestion (eval-set authoring). **Status:** fixed in both cases (eval-018 rewritten to require multi-step branch reasoning rather than a table lookup; eval-023 reworded to specify which side, with a new companion item, eval-049, added for the other branch). Worth carrying forward as a verification habit for Task 5.1's own harness and any future hand-written eval items: check the scenario's plausibility against the grade's other constraints (overs caps, time windows), not just whether the cited section and number are real.

## Baseline evaluation run (2026-07-13/14)

The following findings came out of building the baseline evaluation harness (`evals/run_baseline.py`, `evals/score.py`, `evals/compile_report.py`) and running it against the full 50-row golden set (66 total generation runs including the approved N-rerun protocol). Full results: `evals/results/baseline_2026-07-13.json` / `baseline_report.md`. Pipeline was frozen for this run — no prompt, tool, retrieval, or judge changes; everything below is either a harness bug (caught and fixed before the real run) or a genuine baseline finding.

## 21. Finding #19 reconfirmed still open; run proceeded on golden-set-is-correct basis

**What happened:** Precondition check before running the baseline: re-checked whether `evals/golden_set.json`'s `eval-028`..`031` (`expected_behavior: web_answer`) and `backend/agent_prompts.py`'s deployed `AGENT_SYSTEM_PROMPT` (F3 paragraph, implements `honest_fallback` for the same category) had been reconciled since finding #19 was logged. They had not — same contradiction, byte-identical in cause. User decision for this run: treat the golden set as correct and the prompt as stale; record the resulting behavior mismatches on these 4 rows as pre-declared expected failures, not new bugs. No prompt change made (pipeline frozen).
**Root cause:** Same as #19 — unresolved.
**Layer:** generation (prompt) vs. project docs. **Status:** still open. The baseline run's own numbers confirm the prediction in #19 exactly: all 4 rows (`eval-028`–`031`) scored `behavior_match=False`, reported as a named subgroup (`general_knowledge_f3_contradiction`) in `baseline_report.md` rather than mixed into general findings. Whoever picks up Task 6 needs to resolve #19 before this subset's numbers mean anything beyond "confirms the known gap."

## 22. ragas==0.4.3 unconditionally imports a langchain-community submodule that no longer exists

**What happened:** `evals/`'s fresh `uv` install pulled `ragas==0.4.3` (current PyPI release) and `langchain-community==0.4.2` (also current, no version ceiling in ragas's own dependency spec). `ragas.llms.base` does a top-level `from langchain_community.chat_models.vertexai import ChatVertexAI` — that submodule was removed in `langchain-community>=0.4` (moved to the standalone `langchain-google-vertexai` package), so `import ragas` fails immediately with `ModuleNotFoundError`, even though this project uses neither Vertex AI nor langchain-community for anything.
**Root cause:** Upstream packaging bug in ragas 0.4.3 — an unconditional import of a module that its own installed dependency (`langchain-community`, unpinned) no longer ships.
**Layer:** evals tooling (third-party dependency), not this project's code. **Status:** worked around, not fixed upstream. `evals/score.py` stubs `sys.modules["langchain_community.chat_models.vertexai"]` with the real `ChatVertexAI` class (from `langchain-google-vertexai`, added as an explicit dependency purely to satisfy this) before any `import ragas`. Contained entirely in `evals/`'s own venv. Worth checking on a ragas upgrade whether this shim is still needed.

## 23. Harness bug: naive blank-line splitting fragmented single retrieved chunks

**What happened:** Caught during a deliberate 3-row smoke test before the full run (not during the real run itself). `evals/run_baseline.py`'s first version extracted each tool's retrieved contexts by splitting on `"\n\n"`, matching how `search_rules_tool`/`web_search_tool` join multiple results together — but retrieved rule-chunk text itself frequently contains internal blank lines (paragraph breaks within one ~500-800 token chunk, per `docs/submission.md`'s chunking strategy), so this silently fragmented single chunks into many spurious separate "contexts" (one row showed 80 "contexts" from 3 tool calls that should have produced ~15).
**Root cause:** The join separator and a plausible internal-content separator were the same string, and the original code didn't disambiguate them.
**Layer:** evals tooling. **Status:** fixed before the real run — splitting now requires the blank line to be immediately followed by a new block's `[Section ...]`/`[title](url)` header (`_BLOCK_BOUNDARY_RE = re.compile(r"\n\n(?=\[)")`), confirmed against real smoke-test output (80 spurious blocks → 10-12 correct ones, verified by eye against the actual tool output format).

## 24. Harness bug: thread_id collision risk on script re-invocation under the same run-date

**What happened:** Also caught during the smoke test. Re-running `run_baseline.py --only eval-002,...` a second time (same `--run-date`, after a code fix) reused the exact `thread_id` string (`baseline-{run_date}-{id}-run{run_index}`) from the first attempt. The LangGraph Postgres checkpointer persists by `thread_id` independent of the harness's own JSONL-based resume logic, so the second invocation silently replayed the first attempt's old conversation state into what should have been an independent run — one row went from "flagged" (first, buggy attempt) to "clean" (second attempt) partly because of this, not only because of the code fix, which briefly looked like a false confirmation of the fix.
**Root cause:** `thread_id` was deterministic from `(run_date, id, run_index)` alone, with no protection against the same script re-processing the same row twice under one run-date (e.g. after a mid-development bugfix, exactly the scenario that surfaced this).
**Layer:** evals tooling. **Status:** fixed — `thread_id` now has a `uuid4` suffix, guaranteeing a fresh thread on every invocation regardless of how many times a given `(run_date, id, run_index)` is re-run. Resumability is unaffected since it's keyed off the JSONL's `(id, run_index)` pairs, not `thread_id`.

## 25. LangSmith was entirely unwired despite being a locked stack decision

**What happened:** Before this session, there was no LangSmith instrumentation anywhere in the codebase — no `LANGCHAIN_TRACING_V2`, no `langsmith` import, nothing — despite `docs/submission.md` listing "LangSmith: traces, cost, evals" as a locked stack decision. Turned on for the baseline run via env vars only (`LANGCHAIN_TRACING_V2`, `LANGCHAIN_API_KEY`, `LANGCHAIN_PROJECT=lexisport-baseline-2026-07-13`) plus `tags`/`metadata` passed through `graph.invoke(config=...)` — zero code changes, since LangChain/LangGraph auto-instrument from these. Confirmed working via the LangSmith API directly (not just assumed): every root run in the session carries the expected tags and metadata, plus an auto-captured `revision_id` matching the actual git commit.
**Root cause:** The stack decision in `docs/submission.md` was never implemented back into any actual running code or deployment env vars (Render's `render.yaml` doesn't set `LANGCHAIN_*` either).
**Layer:** project docs vs. deployment config — another case of the same pattern as #19 (a documented architecture decision that was never wired up). **Status:** open for anything beyond this one baseline run. If Task 6 or production should have tracing on by default, `LANGCHAIN_TRACING_V2`/`LANGCHAIN_API_KEY`/`LANGCHAIN_PROJECT` need to be added to Render's env vars for `lexisport-agent-api`, not just set ad hoc for eval runs.

## 26. Baseline result: table_lookup category is the weak point, judge/Ragas agree only 57% of the time

**What happened:** Aggregate baseline numbers (n=66 generation runs, gpt-4o-mini, local invocation against the real proxy+DB): `answer` rows mean faithfulness 0.749, context_precision 0.708, context_recall 0.797, answer_relevancy 0.655. By category, `table_lookup` (13 rows, includes the Appendix-B-style multi-step questions) is a clear outlier: 31% behavior_match, faithfulness 0.47, and 85% `needs_human_review` — far worse than `direct_rule_lookup` (94% behavior_match, faithfulness 0.82) or `formula_arithmetic` (100% behavior_match, though still only 0.70 faithfulness). Separately, our own judge's PASS/FAIL and Ragas's faithfulness (binarized at >=0.8) agree on only 17/30 (57%) of `answer` rows with both signals present — confusion matrix: 13 judge-pass/ragas-pass, 10 judge-pass/ragas-fail, 3 judge-fail/ragas-pass, 4 judge-fail/ragas-fail.
**Root cause:** Not diagnosed here by design — this was a baseline measurement run only, root-causing and fixing is separate follow-on work. Recorded as the headline weak-number finding this baseline exists to produce.
**Layer:** generation (table/multi-step reasoning specifically) plus judge calibration (the judge-vs-Ragas gap). **Status:** documented-for-Task-6, not fixed. Full per-row detail in `evals/results/baseline_report.md`.

## 27. Two silent content-fabrication failures reproduced identically on both local invocation and the live deployed API

**What happened:** A 6-row HTTP sanity sample against the live Render `/api/chat` (added specifically to check whether local-invocation baseline data represents deployed behavior) reproduced two findings from the main local-invocation run byte-for-byte in kind, not just coincidentally similar:
- `eval-023` (Senior Women's fielding-short-of-overs penalty): both the deployed API and local invocation confidently cited "5 penalty runs" from an unrelated section, instead of the correct 36 runs via Section 5.3.5 (3 overs short × 12 runs/over). Our judge PASSed the draft on both systems — Ragas faithfulness on the local run was 0.33.
- `eval-033` ("What time do matches start on round 1?"): both systems answered with real-world ICC/T20 World Cup fixture information instead of asking which MYCA grade — this is the exact failure mode `golden_set.json`'s own notes for this row already named as a known risk ("the exact phrasing pattern that misfired to web_search_tool during earlier live testing"), now confirmed still live in production on both paths, not fixed by anything since that note was written.

Building the baseline report's "10 worst rows" ranking also surfaced a self-inflicted bug worth recording: a first version ranked judge-*flagged* rows as worse than unflagged-but-wrong rows (backwards — an unflagged wrong answer reaches the user with no warning at all, which is strictly worse). Fixing it naively (treat any `behavior_match=False` as worse than any flagged row) over-corrected: it conflated routing/category mismatches (e.g. an honest fallback where the golden set expected a real answer) — which the judge was never scoped to catch in the first place — with genuine unflagged content fabrication, and the routing mismatches drowned out the real signal. Final version narrows "silent failure" specifically to low/unscoreable Ragas faithfulness on an `answer`-category row that the judge didn't flag; routing/category mismatches get their own, lower-severity tier.
**Root cause:** `eval-023` and `eval-033` are genuine generation-layer failures (wrong-section citation; a "round 1" phrasing that still gets misread as a global-tournament reference despite `golden_set.json` already documenting the risk) — not deploy/local configuration drift, confirmed by reproducing on both paths.
**Layer:** generation (both cases) plus judge calibration (`eval-023` specifically — a real error the judge's own faithfulness-style check should plausibly have been able to catch and didn't). **Status:** documented-for-Task-6. Also note: the HTTP sanity sample initially failed all 6 calls with `401 Unauthorized` because the local `.env`'s `API_KEY` differs from Render's deployed `API_KEY` (expected — `render.yaml` sets it `sync: false`, i.e. independently per environment — but worth knowing this blocks the sanity script until the correct deployed key is supplied at invocation time, never written to a file).

## Hybrid retrieval experiment (2026-07-14)

## 29. Mid-run Postgres connection drop exposed a resumability gap across all three harness scripts

**What happened:** The hybrid generation run's Postgres connection died mid-run (`OperationalError: server closed the connection unexpectedly`, after a 1299-second hang on one call — consistent with an idle-connection timeout on Supabase's pooler during a single long-lived `PostgresSaver` connection held open for the whole ~15-20 minute batch run), cascading into 5 consecutive error records for the rows still to come. On investigating the fix, found the resume logic in `run_baseline.py` treated *any* existing JSONL line for `(id, run_index)` as "done," including error records — a resumed run would have skipped retrying the failed rows forever. None of `run_baseline.py`, `score.py`, or `compile_report.py` deduped by `(id, run_index)` on read either, so a retried row appending a second line for the same key would have been double-counted in every downstream aggregate.
**Root cause:** The append-only/resumable design (built for the baseline run, which never hit an error) was never actually exercised against a real error-and-retry case until this run hit one live.
**Layer:** evals tooling. **Status:** fixed in all three scripts — resume logic now only treats a key as "done" if its *last* record has no error (so errored rows retry automatically), and all three readers now apply last-record-wins dedup by `(id, run_index)` before using the data. Verified live: resumed the hybrid run, all 5 previously-errored rows completed successfully, final coverage confirmed at exactly 66 unique `(id, run_index)` pairs with zero residual errors.

## 30. Hybrid retrieval does not beat baseline in aggregate

**What happened:** Full 50-row/66-run comparison, identical N-run protocol, generator, judge, and prompts as baseline — retrieval mode was the only variable. Net negative: 11 rows regressed (clean at baseline, worse at hybrid) vs. 8 improved, and most aggregate metrics moved down — overall behavior_match 60.6% → 57.6% (-3.0pp), context_precision 0.708 → 0.656 (-0.052), context_recall 0.797 → 0.777, answer_relevancy 0.655 → 0.63, needs_human_review rate 33.3% → 39.4% (+6.1pp worse), judge PASS rate 66.7% → 60.6% (-6.1pp). **Layer:** retrieval (the real component of the regression) plus judge/Ragas calibration (a confound baked into how the comparison is measured, not itself changed by this experiment). **Status:** decision made from these numbers — keep baseline, hybrid not shipped. `RETRIEVAL_MODE` was never set on Render, so no deploy or rollback was needed either way.

## 31. eval-023: hybrid retrieval produced the objectively correct answer and scored worse than baseline's wrong one

**What happened:** The most important single-row result of the experiment, and it complicates finding #30's aggregate numbers directly. Under hybrid, `eval-023` achieved context_recall 1.0 (up from baseline's 0.0 on this same row) and the model's final answer was "36 penalty runs (Section 5.3.5)" — matching the golden set's verified ground truth exactly, word for formula ("12 penalty runs per over × 3 overs = 36"). Baseline's answer on the same row was confidently wrong ("5 penalty runs," an unrelated disciplinary-sanctions section). Yet the *scores* went the opposite way: baseline's wrong answer PASSED the judge silently (faithfulness 0.33, a known silent-failure case, see finding #27); hybrid's objectively correct answer FAILED the judge (faithfulness 0.0) — because hybrid's full-text leg pulled MYCA's "LEVEL 3/4 IN-GAME SANCTIONS/PENALTIES" disciplinary sections to the top of retrieval on the literal word "penalty," and both the judge and Ragas appear to have been misled by that co-retrieved disciplinary content into judging the correct "12 runs/over" claim as unsupported ("the evidence specifically states 5 runs per over") — even though the generator itself was not fooled by the same context.
**Root cause:** The full-text leg's literal keyword matching doesn't distinguish "penalty runs" (a playing-conditions term) from "penalties" (the disciplinary code's term) — the exact vocabulary-collision risk hybrid retrieval was designed to reduce turns out to also apply in reverse, contaminating context with lexically-similar-but-topically-wrong sections. That contamination didn't fool the generator on this row, but it did fool both automated evaluators.
**Layer:** retrieval (the vocabulary-collision itself) plus judge/Ragas calibration (both evaluators scoring a correct answer as wrong because of noise in the surrounding context, not the claim itself). **Status:** documented, not fixed. Directly informs finding #33's recommendation (down-weight or filter the full-text leg) and is a caution against reading this experiment's aggregate metrics as a clean proxy for actual answer quality — a human reading the raw answer text would call hybrid's `eval-023` result a clear win, not the regression the scores say it is.

## 32. eval-015: the retrieval hypothesis didn't hold for its own strongest motivating row

**What happened:** `eval-015` was one of the two rows (with `eval-023`) used to motivate this experiment (finding #28). Under hybrid, the failure mode was unchanged: both baseline and hybrid confidently pick one interpretation of the ambiguous question instead of asking a clarifying question or presenting the relevant branches — the judge's rationale text is nearly word-for-word identical between the two runs ("confidently presents/states a single interpretation... without addressing the ambiguity..."). What actually got retrieved differed between modes, but it didn't matter: the failure is a generation-layer disposition (the model doesn't hedge on ambiguous multi-grade questions), not a retrieval gap, for this specific row.
**Root cause:** Retrieval quality was never the bottleneck for `eval-015`'s observed failure mode — improving retrieval cannot fix a generation-layer unwillingness to ask a clarifying question.
**Layer:** generation. **Status:** documented-for-future-work — a real, separate problem from anything this retrieval-only experiment could address by design.

## 33. Naive full-text fusion (1/1 weights, unfiltered) is a double-edged sword worth re-tuning, not a settled failure

**What happened:** Across the row-level reads (`eval-016`, `eval-023` especially), the pattern is consistent: `match_rule_chunks_hybrid`'s full-text leg matches on short, common words that appear in both the correct playing-conditions rule text and unrelated administrative/disciplinary sections ("penalty," "balls," "bowled"), pulling wrong-topic chunks into the fused top-k with real weight. This is the same mechanism that made the *dense* leg miss `eval-023` in the first place (semantic drift toward "penalties"/sanctions), now showing up as a *literal* collision on the full-text side instead.
**Root cause:** Default RRF weights (1/1) and no content-type filtering or stopword/domain-term tuning on the full-text leg — the simplest possible configuration, not a tuned one.
**Layer:** retrieval. **Status:** open, recommended as the next iteration if hybrid retrieval is revisited: down-weight the full-text leg relative to semantic (e.g. `full_text_weight < semantic_weight`), or exclude `content_type`/section categories like disciplinary codes from the full-text index, before re-running this comparison. Per the project's own one-variable-at-a-time rule, this would be a new, separate experiment, not a patch to this one.

## 35. One tuning round: 7 false-fires traced to 4 concrete bugs, fixed

**What happened:** The first full golden-set run (2026-07-14) surfaced 7 rows clean at baseline that the mechanical check newly flagged. Traced to 4 distinct bugs, not vague over-firing: (1) `citation_check_node` reused `judge_node`'s `_gather_tool_evidence`, which only reads `ToolMessage` content — `retry_retrieval_node` injects its broadened evidence as a `SystemMessage` (retry bypasses `ToolNode` entirely), so the check was evaluating post-retry drafts against stale, pre-retry evidence (6 of 7 false-fires, all with `retry_count == 1`); (2) web citations in a `"(Source: [Title](URL))"` nested-markdown-link format weren't parsed (`eval-010`); (3) the evidence-header regex mis-split on section headings containing their own internal comma, e.g. `"5.1 Naming of teams, the toss and commencement of play"` (`eval-040`, `eval-045`); (4) the dotted-prefix containment guard required the retrieved (parent) code to itself contain a dot, which incorrectly blocked legitimate bare top-level chapter headings like `"15 Blood Rule"` (real sub-clauses 15.1-15.6) from matching a citation to `"15.4"` (`eval-048`).
**Root cause:** (1) is an evidence-gathering gap in the new node itself (fixed in `backend/agent.py` via a dedicated `_gather_citation_evidence`); (2)-(4) are parser/matcher gaps in the new module (fixed in `backend/citations.py`) — see that module's own docstring for full technical detail.

## 36. Two known limitations documented, not patched — the one-tuning-round line held

**What happened:** After finding #35's fixes, 2 false-fires remained. `eval-040`: two real, correctly-retrieved sections cited together in one compound parenthetical (`"Section X, Doc A; Section Y, Doc B"`) — the module deliberately refuses to auto-split this ambiguous shape rather than guess, the same conservative choice already made for `eval-027`'s identical format. `eval-045`: cites `"Section 11.8.6"`, which genuinely exists verbatim in the retrieved evidence, but as a sibling clause appended after `"11.8.5 Appeals Tribunal Members"` in the same bundled chunk, not a dotted-prefix child of it (`"11.8.6"` doesn't start with `"11.8.5."`) — the containment matcher only recognizes parent-child relationships, not adjacent siblings sharing one bundled chunk.
**Root cause:** `eval-040` is an accepted design trade-off (fail-toward-review on ambiguous compound citations). `eval-045` would require searching a chunk's literal body text for the cited number rather than comparing structural section labels — a real capability extension, not a normalization tweak.
**Layer:** mechanical check (`backend/citations.py`). **Status:** documented as known limitations, pinned with dedicated unit tests (`test_unparseable_compound_citation_fails_toward_review`, `test_known_gap_sibling_clause_in_bundled_chunk_is_not_matched`), not fixed — per the approved plan's own discipline, a second tuning round would mean the citation format is less controlled than assumed, which is a finding, not another patch.

## Judge evidence-gathering and verdict-normalization experiment (2026-07-16)

Two independent judge-side fixes were built and evaluated together against the `citecheck_2026-07-15` anchor, both aimed at the same symptom: the judge FAILing a turn while leaving `unsupported_claims`/`fabricated_citations` empty, a structured-output gap where its prose reasoning doesn't reach its structured fields. (1) Evidence-gathering: `judge_node` was still using a ToolMessage-only gatherer (`_gather_tool_evidence`), missing retry_retrieval_node's broadened-search results the same way `citation_check_node` did before finding #35 — unified into one `_gather_turn_evidence` used by both nodes. ## 40. Verdict normalization silently suppressed a correct FAIL the golden set says should exist

**What happened:** Only 2/66 rows actually triggered `verdict_normalized` in the combined run. `eval-004` was benign — its reasoning named a genuinely missing detail, golden `expected_behavior` is `answer`, and `behavior_match` stayed `True` whether normalized or not. `eval-047` was not: its golden `expected_behavior` is `flag_acceptable`, and the golden set's own notes call this out explicitly — the question ("What does the Executive Committee do?") is broader than its single-section source grounding and a genuine answer would need to pull from many sections, so a flag is the *correct* outcome, not a false positive. The judge's FAIL reasoning independently identified exactly that problem in prose. But `unsupported_claims` and `fabricated_citations` were both empty, so the normalization rule silently converted this correct FAIL into a PASS — removing a flag the golden set says should exist, on a row where the judge had actually gotten it right.
**Root cause:** The normalization rule inferred "no problem" from empty structured-output arrays, but an empty array here meant the judge's finding didn't fit the `unsupported_claims`/`fabricated_citations` shape (a scope/coverage judgment, not a per-claim or per-citation one), not that there was no finding. Treating "doesn't fit the schema" as "no problem" is exactly backwards for a safety check.
**Layer:** judge (`backend/agent.py`, `_normalize_verdict` — since removed). **Status:** caught before shipping by per-row VERIFY tracing, not by a user or a later audit — this is what the discipline is for. See finding #41 for the decision this led to.

## 41. Normalization dropped, evidence-gathering kept: a false negative outweighs the false-positive noise it was built to remove

**What happened:** Given finding #40, three options were on the table: (a) narrow the normalization heuristic further, (b) ship it with eval-047 as a documented known limitation like the citation check's two accepted gaps (findings #36, #38), or (c) drop normalization entirely and keep only evidence-gathering. Chose (c). (a) is still a heuristic with its own blind spots and would need its own tuning round before trusting it, the same one-tuning-round discipline finding #36 already leaned on. (b) doesn't fit: the citation check's known limitations are cases it was never designed to catch (compound citations, sibling clauses); eval-047 is the opposite — a case normalization was actively designed to touch, and it suppressed a correct catch, not a documented boundary. Re-ran the full 66-row VERIFY with normalization removed and evidence-gathering alone: needs_human_review 45.5% → 34.8% (-10.7pp) and judge PASS rate 54.5% → 65.2% (+10.7pp) vs. the `citecheck_2026-07-15` anchor (`evals/results/comparison_evidencefix_2026-07-16.md`) — evidence-gathering alone keeps 10.7 of the combined run's 16.7pp improvement (64%); normalization's own marginal share was the other 6.0pp, and that share is exactly what was carrying eval-047's suppression. `eval-047` now correctly stays flagged (`needs_human_review=True` on 2 of 3 N-runs, vs. 0 of 3 with normalization in place); `eval-004` is unaffected, PASSing on its own merits with no normalization needed. The 4 rows that regressed vs. the citecheck anchor in this rerun (`eval-001`, `eval-019`, `eval-026`, `eval-039`) are a different set from the combined run's regression list — checked each individually: three (`eval-001`, `eval-019`, `eval-039`) show `citation_verdict` PASS with `judge_verdict` FAIL (pre-existing judge nondeterminism, the same class as finding #37), and one (`eval-026`) is a `behavior_match` flip with both verdicts PASS on both runs (generation-wording nondeterminism, not a verification-layer change). None involve the mechanical check newly firing, and none could be normalization-caused since that code path no longer exists. The mechanical catch row-membership also shifted (`eval-011`, `eval-045` dropped out; `eval-015`, `eval-042` newly appeared) — `backend/citations.py` and `_gather_turn_evidence` are byte-identical to the combined run, so this is draft-text variance between separate live runs, not a code effect.
**Root cause:** N/A — a decision record, not a bug. The actual root-cause fix — requiring the judge to always populate `unsupported_claims`/`fabricated_citations` on a FAIL, so "no problem named" can never happen regardless of what caused the FAIL — is a `JUDGE_SYSTEM_PROMPT` change, out of scope for this task, deferred to next-steps.
**Layer:** judge (`backend/agent.py`). **Status:** shipped — `_normalize_verdict` removed, `_gather_turn_evidence` kept. Every checker in this system fails toward review when uncertain; a normalization rule that silently passes a correct catch breaks that invariant, so it doesn't ship even though it costs most of the headline number.

### Baseline Evaluation Findings (originally in submission.md, against the 14-question table before the 50-row golden set existed)

I ran the baseline retrieval and answer pipeline (search_rules, then one gpt-4o-mini call through the LiteLLM proxy, no agent framework, no judge, no web search) against real questions, using the full MYCA corpus that is actually ingested: Junior, Senior Men's, and Senior Women's playing rules, plus the two operational forms. This section replaces guesswork with what the system actually did.

The first finding is about the evaluation table itself. Of the ten expected answers I have now checked against real documents, five turned out to be wrong. Question 1 assumed a flat one over per four minutes formula and stated 34 overs, but the real rule for a non-weather late start points to a separate lookup table, and 6 minutes lost falls in that table's 5 to 8 minute band, which is 2 overs, giving 33 overs, not 34. Question 4 invented a run rate based revised target method. The real rule is simpler: a lost result after play has commenced is just a draw. Question 6 assumed MYCA has a powerplay system that differs by grade. I searched every ingested document for the word powerplay and found it nowhere. No such system exists at all. Question 7 assumed a single rule ties a transfer cut-off date to finals eligibility. The real rules keep those two things separate, a one-transfer-per-season limit and a games-played threshold for finals, with no cut-off date connecting them. Question 14 invented a specific sightscreen rule. No such rule exists, only a general boundary six rule for a ball hit into a tree or other obstacle. Only questions 2, 5, 9, and 10 turned out to be correct as originally written. I wrote all fourteen expected answers before the real documents were ingested, and half of the ones I have now checked were wrong, in one case, question 6, wrong about a whole rule existing at all. An eval table written before a corpus exists is a guess, not a ground truth.

The second finding is about retrieval recall. In three cases, question 1, 4, and 14, the real answer existed somewhere in the ingested chunks, but the specific clause needed, a lookup table, a two-line clause in a five-sentence section, or a deep sub-clause, did not make the top 5 results. The baseline did not guess in any of these cases. It either said the rules do not appear to cover the question, or, for question 1, correctly named the applicable rule and then honestly admitted it could not state a number.

The third finding is that even when the right chunk is retrieved, the model does not reliably use it. Question 13 is the clearest case: the exact clause needed, a female age allowance to play in a boys' competition, was retrieved above the relevance threshold and included in the prompt, and the model still said the rules do not cover the question. I saw the same pattern testing two multi-step calculations directly outside the table. A question about time lost during a first innings produced three correct answers and one wrong one across four attempts with the same retrieved context and no fixed temperature setting. A related question about a mid-innings weather delay was refused three times in a row, even though the exact formula it needed was sitting in the context it had already retrieved. Retrieval itself was reliable throughout. The single LLM call is not, in both directions: sometimes refusing an answerable question, sometimes making an arithmetic mistake partway through a multi-step rule.

The fourth finding came from ingesting the rest of the corpus today. With only the Senior Men's document ingested, there was no way to notice this, but match_rule_chunks originally filtered only on association_id, not grade. Once Junior and Senior Women's documents shared the same association_id, a senior-grade question could in principle retrieve and cite a junior-only or women's-only chunk as if it applied. Question 8 showed a real version of this: asked without specifying a grade, the baseline confidently quoted Senior Men's forfeit fine, $100 to $200 by round, with no mention that Senior Women's uses a flat $50 fine instead. I fixed the underlying gap today, match_rule_chunks and search_rules now take an optional grade_scope filter, verified working, but the fix only helps when a grade is actually supplied. A genuinely grade-ambiguous question, like question 8's, still needs either a clarifying question back to the user or an explicit warning that the answer may vary by grade, and the baseline does neither.

The fifth finding came from a question outside the table, once both operational forms were ingested. Asked what the process is if a bowling action is found to be illegal, the baseline retrieved both the general rules text on suspect actions and the dedicated Suspect Bowling Action Form's own procedure chunk, both above the relevance threshold. The answer used only the rules text: correctly describing what happens once a report escalates, no action on the day, the club must rectify, a second report suspends the player from bowling, a Cricket Victoria remedial session before they can return. It never mentioned the form's own specific first step, submit that exact form to the Secretary and Umpires' Coordinator within 48 hours, keep it confidential, one form per player. Both chunks were sitting in the same context. The model favored the rule_text chunks over the procedure chunk and produced an answer that was accurate as far as it went but left out the actual first action an umpire needs to take.

None of this failed silently in the sense that matters most. Every fallback I saw named no invented rule number and no fabricated section, the one case where a wrong number reached an answer, question 8, was a real number from a real document, just the wrong grade's, not an invention, and the suspect-bowling case was accurate as far as it went, just incomplete. That is the property I care most about holding at this baseline stage. It held in every test except the grade-ambiguity gap, which is now at least partially addressed at the retrieval level.

### Agent Wrapper Findings (originally in submission.md)

I wrapped the frozen baseline's search_rules function in a LangGraph agent: the model decides when to call it as a tool, instead of the baseline's fixed retrieve-then-answer, plus conversation memory through the LangGraph Postgres checkpointer. This is a different component from the baseline above, and it surfaced problems the baseline never could, since the baseline always retrieved unconditionally and never chose a grade.

The first finding is that giving the model control over tool calling introduced two new failure modes that have nothing to do with the baseline's documented weaknesses. I ran the 33-minutes question seven times before making any changes. One run never called the tool at all, the model decided the question was "a calculation based question... rather than a question about the specific rules" and answered from that false distinction alone. Three more runs called the tool but left grade_scope unset, and retrieval, searching across all four grades in the corpus now that Junior and Senior Women's are also ingested, returned the Junior document's daily overs rule for a question about a senior match, and the model answered from the wrong grade's numbers without noticing. Only three of seven runs actually retrieved from the right document.

The second finding is that these two failures needed two different kinds of fix, and only one of them belonged in a prompt. The tool description had left an opening for the model to treat a calculation as somehow separate from a rules question, so I rewrote it to say plainly that a question needing arithmetic still needs the tool first, to find the rule and its numbers. That is wording tuning, and it is what this stage of the project is for. The wrong-grade problem is not a wording problem. It is the same cross-grade contamination that match_rule_chunks was built to prevent at the SQL level last session, reintroduced one layer up by letting the model choose grade_scope itself. I removed grade_scope from the tool's schema entirely and made it a caller-supplied parameter, exactly like association_id: closure-captured at graph-build time from a trusted source, the CLI's --grade-scope flag today, invisible to the model, and unchangeable by anything the model outputs. I confirmed this by printing the tool's actual JSON schema before and after, association_id was never there, and after the fix, grade_scope was not there either.

The third finding is that both fixes held up under repeat testing. Five more runs of the same question after the fixes called the tool five times and retrieved from the correct grade five times. This is a small sample, the same size as the sample that first showed the problem, but the change from three-out-of-seven to five-out-of-five on the same question is a large enough swing that I am treating it as a real fix rather than noise, while still keeping in mind that five runs is not proof of five hundred.

The fourth finding is that memory works, and I have direct database evidence rather than just plausible-looking output. Two turns in the same conversation thread, asking about a rule and then asking what happens if a team falls short of it, resolved the pronoun correctly on the second turn, and produced ten chained checkpoint rows and fourteen checkpoint_writes rows in Supabase, each checkpoint pointing at its parent. The same second-turn question asked on a fresh thread produced a completely different, sensible answer with no memory of the first thread at all, and the model flagged the ambiguity itself rather than guessing which meaning I intended. I also tested persistence across two entirely separate CLI processes, not just two calls in the same script: the first process exited completely before the second one started, and the second process still resolved a follow-up question against the first process's turn. That is only possible if the state lives in Postgres, which is what the checkpointer is for, not in memory that a process holds and loses on exit.

The fifth finding is about what carried over from the baseline unchanged. Association isolation held through the agent exactly as it held through the baseline: a question against an association with zero ingested chunks called the tool with that association's real id, retrieved nothing, and produced an honest fallback rather than leaking another association's content. The arithmetic unreliability documented in the Baseline Evaluation Findings above did not go away, since I deliberately left the grounding and citation wording in the system prompt unchanged from the baseline to keep that comparison valid, and the same 33-minutes question that the agent now reliably retrieves the right chunks for still sometimes gets the actual calculation wrong once it has them. That is expected, and it is exactly why I did not touch that wording here. Fixing generation quality is separate work from fixing whether the right tool gets called with the right scope, and conflating the two would have made it impossible to tell, later, which layer any given wrong answer came from.

### Web Search Tool Findings (originally in submission.md)

I added web_search_tool as the agent's second tool, wrapping the Tavily API directly rather than a prebuilt LangChain integration, so the routing description, the failure handling, and the result shaping stay fully in my own words and my own code. This section covers what happened when the agent had a real choice between two tools instead of one.

The first finding is that the model does compose both tools when a question genuinely needs it, and I saw this happen every time it should. A question naming Cricket Australia's concussion substitute rule and asking whether it applies to this grade called both search_rules_tool and web_search_tool in three out of three trials, checked MYCA's own rules first, found they do not explicitly address it, then pulled Cricket Australia's current guidelines and cited the source. This is not a case where the eval table's original Tavily-only expectation was fully right: the question has a real rules-application component, whether MYCA has adopted an external policy, and answering it well needs both sources, not one. Questions that cleanly belong to one side stayed clean: the 33-minutes question stayed on search_rules_tool only across three trials, a Saturday rain forecast stayed on web_search_tool only across three trials, and a plain greeting called neither tool.

The second finding is a real failure, and a different shape from anything the baseline or the memory-and-tool-choice work surfaced before. I asked whether an umpire can reverse his decision, a genuine Laws-of-Cricket question that is in neither MYCA's documents nor, really, the kind of current public information web_search_tool is described for. In three out of three trials the model never called web_search_tool at all. It called search_rules_tool, retrieved a tangentially related chunk about umpires being sole judges of light and weather, and then added an uncited claim along the lines of general cricket practice allowing umpires to reverse a decision on reflection. That sentence has no citation, and it should not exist at all under the prompt's own rule against answering from general knowledge. Two of the three trials at least flagged first that the retrieved rules do not mention decision reversal before adding the unfounded claim anyway, which is more honest in form but not in substance, since the fabricated sentence is still there either way.

The third finding is that this second failure points at a real gap in the two-tool design itself, not just a prompt-following slip. search_rules_tool is scoped to the association's own documents, and web_search_tool's description is about current public information, fixtures, results, news, weather, other organizations' current policies. A timeless Laws-of-Cricket question that is not association-specific and not current fits neither description cleanly, and the model filled that gap with speculation instead of an honest fallback. The base-Laws case is exactly what the Data Strategy section already named as needing the model's own general cricket knowledge together with Tavily, but the current system prompt's grounding rule still says not to answer from general knowledge, and nothing routes this third category of question anywhere deliberately. That is a scope boundary this task's tools do not resolve, and I have left it that way rather than patching it here.

The fourth finding is a characterization, not a pass or fail, of where the routing line falls for questions that name an external body without a clean split like the concussion case. Asked whether Cricket Australia's code of conduct overrides the association's own conduct rules, the model called search_rules_tool only, in three out of three trials, and answered from MYCA's own rule that members are bound by the Laws of Cricket and the association's rules. It treated this the same way it correctly treats the MCC over-count divergence question in the eval table, as something the local rules resolve on their own without needing to know the external body's actual current content. Two of the three trials were honest that the local rules do not explicitly settle the override question. None of the three reached for web_search_tool to actually check Cricket Australia's current code, even though the tool description permits calling both tools for exactly this kind of question. Whether that is the right default or not is a real design question for later, not something I am resolving here.

The fifth finding is that memory and the two-tool routing work together correctly. A two-turn conversation asking a rules question about boundary distance, then a follow-up asking about current news for the association's specific ground, resolved the boundary figure from turn one and correctly used both tools on turn two, with an honest fallback for the part no news coverage existed for. Ten checkpoint rows were written for that thread, matching the count from the single-tool memory test in the Agent Wrapper Findings above, so adding a second tool did not change how memory is recorded.

None of these findings change what I already knew about citations and honest fallbacks from the baseline and the first agent-wrapper pass. What is new here is that a second tool creates a third possible failure, not just wrong-tool and wrong-scope, but a question that fits neither tool's description and gets filled with uncited speculation instead of an honest gap. That is a real, repeatable finding, not a one-off, and it belongs to the judge node's job, not this one.
